UNOES Dictionary Live 1st service / 模索期

AI ツールを乗り換えながら、約1年かけて本番投入したマルチプラットフォーム辞書アプリ

UNOES は私にとって最初の本格的な AI 駆動開発プロジェクトです。 2025年4月〜2026年4月、中断期間を含むおよそ1年間で、 iOS / Android / Web / Windows をターゲットにした辞書アプリと Go バックエンドを個人開発しました。 「AI 駆動開発はどう進めるべきか」を答えのないまま模索した期間でもあり、 ツールを乗り換えるたびに開発スタイルが大きく変わっていきました。

Phase 1 2025/04 – 2025/夏
ChatGPT（コード生成→コピペ）

仕様や設計の壁打ち、コード片を ChatGPT に書かせてエディタに貼る、というスタイル。手戻りも多く、コードベース全体の文脈は AI 側に残せませんでした。

Phase 2 2025/夏 – 2025/秋
Cursor AI（IDE 内 AI 補助）

エディタ統合の AI に切り替え、コードベース全体を参照させながら編集できるようになり生産性が大幅向上。ただしレビュー観点は属人化したまま。

Phase 3 2025/秋 – 2026/04
Claude Code（エージェント駆動）

Claude Code に移行し、CLAUDE.md / memory にプロジェクト固有ルールを残し、機能ごとに SPEC → PLAN → TEST → IMPL → REVIEW を Markdown で蓄積する現在のスタイルを確立。

この模索期に得たのは「AI に書かせる」ではなく「AI と一緒に意思決定を文書で残す」開発スタイルでした。 次の GIVErS では最初からこのワークフローを適用しています。

  • JWT + リフレッシュトークン認証（admin / editor / subscriber / guest ロール）
  • Stripe / Google Play 両プロバイダーのサブスク（解約 / 期限切れ / ロール降格を一元管理）
  • IP 単位 + 経路別レート制限による DOS 対策ミドルウェア
  • 辞書エントリの4段階編集ステータス（draft / review / published / archived）とフィルタ
  • 全操作の監査ログとフィードバック機能
  • Web 版 Google ログイン（admin / editor 向け）、AdSense ネイティブ広告、リワード広告
  • Flutter から iOS / Android / Web / Windows へワンソース配信

GIVErS Live 2nd service / 確立後

ゼロ手数料の寄付プラットフォーム — Claude Code で約2週間立ち上げ

UNOES で確立したワークフローを最初から適用し、最初の1行から Claude Code と共に書き始めた2本目のサービス。 企画・設計・実装・本番デプロイ（VPS / TLS / DB / Stripe Connect）までを約2週間で立ち上げ、現在は givers.work で稼働中です。 「AI 駆動開発の凄み」を強く実感した体験で、UNOES の1年が嘘のような速度感でした。

※ サービスとしてはまだ未完成で、機能の磨き込みは継続中です。

プロダクトのコンセプトは「クリエイターはGIVEする・利用者は応援したくなったらGIVEで返す」というゼロ手数料寄付プラットフォーム。 プロジェクトの月額コストと達成率を透明に表示することで、寄付者が安心して判断できることを目指しています。 ゼロ手数料を実現するために Stripe Connect (Express) を採用し、プラットフォームを介さずクリエイター本人へ直接送金されます。

  • OAuth2 ログイン（Google / GitHub / Discord）+ セッション認証
  • Stripe Connect による直接送金 / 単発 & 定期寄付 / Webhook 連携
  • プロジェクトのコスト構造と達成率の可視化、マイルストーン通知
  • i18n（日本語 / 英語）対応、Astro Islands + React で SSR / CSR を最適化
  • Playwright による E2E テスト（auth / donation / webhook / admin など全12スイート）
  • ConoHa VPS + Docker Compose + Nginx で本番運用、TLS 自動更新

AI 駆動開発ワークフロー

UNOES の1年で「コピペ → IDE 補助 → エージェント駆動」と AI ツールを乗り換えて辿り着いた、 仕様・計画・テスト・実装・レビューを Markdown で残しながら AI と協働するワークフロー。 リポジトリ内に各機能の議事録が時系列で並び、将来の自分・他の開発者・AI 自身が同じ文脈に追いつけるようにしています。 GIVErS ではこのワークフローを最初から適用し、約2週間で本番投入できました。

  1. 001

    SPEC — 仕様確定

    ユーザー要望（自分自身）を AI が整理し、明確化した仕様を表形式で記録。確認状況（Confirmed / Pending）まで明示。

  2. 002

    PLAN — 実装計画

    影響範囲・追加/変更ファイル・段階分けを AI とすり合わせ、設計判断は1つずつ「推奨＋理由」付きで決定する。

  3. 003 / 004

    UNIT_TEST / E2E_TEST — テスト設計

    実装前にテストケースを Markdown 化（TDD）。Playwright（Web）と Go test（バックエンド）で自動化。

  4. 101 / 102

    IMPL_REPORT — 実装レポート

    AI が実際に行った変更と判断理由を残す。GREEN 後に i18n / セキュリティ準拠を機械的にスキャン。

  5. 201

    FEEDBACK_REVIEW — 利用者フィードバック

    実機で確認した上での不具合・改善要望を AI と整理し、原因と修正範囲を文書化。

  6. 901

    REVIEW — 振り返り

    機能ごとに学びと再発防止策を CLAUDE.md / Memory に還元し、次のサイクルへ。

ワークフローを支える小さな工夫

  • CLAUDE.md / memory — プロジェクト固有のルール（i18n 準拠レビュー / デバッグログ方針 / 1問1答原則 等）を AI に常時参照させる
  • 機能フォルダ分割 — docs/<機能名>/001..901 の規約で、機能単位の文脈を1ディレクトリに閉じ込め
  • 段階レビューゲート — SPEC レビュー → PLAN レビュー → 実装 → REVIEW で stop the line を徹底
  • 機械チェック — 日本語ハードコード抽出、未使用コード検出、ビルド/型/lint を grep / script で自動化

About
